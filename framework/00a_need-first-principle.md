# 00a Need-First Principle

Need-First Principle は、IYASAKA が AI 実装経験を事業価値に変えるための判断原則です。

IYASAKA は AI を深く触っていることを強みにします。
ただし、その強みは、AI に詳しくない人や会社を評価するためのものではありません。
まだ AI を深く触れていない人、会社、現場には、それぞれの事情・優先順位・制約があります。

Rasen は、その相手に対して、AI 実装経験を役に立つ形へ変換するための Growth OS です。

## Core Statement

```text
AIを深く触っているから偉いのではない。
AIを深く触っているからこそ、まだ触れていない人の役に立てる。
```

## Purpose

Rasen は、すごいものを作るための仕組みではありません。
必要とされるものを、必要としている人に、届く形で作るための仕組みです。

AI、MCP、エージェント、RAG、API連携、SaaS化は手段です。
顧客が求めているのは、技術名ではなく、次のような結果です。

- 判断できる
- 説明できる
- 作業が減る
- 失敗しにくくなる
- 現場で回る
- 売上につながる
- 不安が減る
- 次の一手が分かる

## Decision Order

Rasen の判断順序は、技術起点ではなく必要起点です。

```text
誰が困っているか
  ↓
何に困っているか
  ↓
どの状態になれば助かるか
  ↓
そのために、AI・自動化・人手・既存ツールのどれを使うべきか
```

## Anti-Pattern: 型から入る罠

IYASAKA は、構造化・フレームワーク化・AI化が得意です。
そのため、次の罠に注意します。

- 先に型を作ってから、顧客を当てはめようとする
- 先に高度な仕組みを作ってから、売り先を探そうとする
- 顧客の言葉より、自分の構想を優先する
- すごいものを作ることを、必要とされることより優先する
- AIに詳しくない相手の現実を十分に見ない
- 反応がない理由を、市場や相手だけの問題にする

この状態に気づいたら、Rasen は必ず次の問いに戻ります。

```text
これは誰の、どの困りごとを、どのくらい楽にするのか。
```

## Result Commitment

相手の要望や希望にフォーカスすることは、言われた通りに作ることではありません。
結果にコミットするとは、以下を両立することです。

- 顧客の言葉をそのまま受け止める
- 顧客がまだ言語化できていない不安や制約を見つける
- 顧客の希望を、実行可能な順番に変換する
- 必要ないものは作らない
- 今すぐ作るべきものと、後でよいものを分ける
- 顧客が実際に使えるところまで落とし込む

## Speed Rule

Rasen で何かを作る時は、以下の順番で速度を決めます。

1. 顧客の困りごとが明確なら、速く小さく出す
2. 顧客層が曖昧なら、作る前にヒアリング・発信・無料診断で確認する
3. 技術的に作れるだけなら、まだ作らない
4. 受注・商談・問い合わせで同じ困りごとが3回出たら、商品化を検討する
5. 顧客が使う場面が見えないものは、MVPではなくメモに留める

## Product Judgment Questions

新しい商品、LP、投稿、診断フォーム、営業資料、納品物を作る前に、必ず確認します。

1. これは誰のためのものか
2. その人は今どんな場面で困っているか
3. その人は何を望んでいるか
4. その人が実際に使える形になっているか
5. これは自分が作りたいものか、相手が必要としているものか
6. どの層に、どの順番で届けるべきか
7. どれくらいの速度で検証すべきか

## Final Rule

```text
技術の深さを、相手への敬意と実用価値に変換する。
```
